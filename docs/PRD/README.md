# PRD Documentation Guidelines

Use this folder for all Product Requirements Documents (PRDs). Keep files consistent by starting from the template in `docs/PRD/_PRD_TEMPLATE_HUL.md`.

## Naming Rules

- Format: `PRD-YYYYMMDD-FeatureName.md`
- Use a concise, PascalCase feature name.
- One PRD per feature or project.

## Required Sections

PRDs must include these sections (in this order):

1. Introduction / Overview
2. Goals
3. User Stories
4. Functional Requirements
5. Non-Goals (Out of Scope)
6. Design Considerations
7. Technical Considerations
8. Success Metrics
9. Implementation Plan
10. Testing Requirements
11. Security & Permissions
12. Deployment Plan
13. Timeline
14. Open Questions & Risks
15. References & Resources
16. Revision History

## PRD Checklist

- [ ] File name follows `PRD-YYYYMMDD-FeatureName.md`
- [ ] Header fields completed (ID, dates, author, status, related scripts)
- [ ] All required sections included, even if brief
- [ ] Acceptance criteria are explicit and testable
- [ ] Script inventory and deployment details listed (when applicable)
- [ ] Open questions and risks captured
- [ ] Revision history updated

## Solo Review (Before Implementation)

- [ ] Problem statement and success metrics are clear to a non-technical reader
- [ ] Acceptance criteria are testable and unambiguous
- [ ] Out-of-scope items are explicit
- [ ] Dependencies and governance risks are captured
